# App Version Single Source of Truth — Implementation Plan

## Summary

Move the app version string from a hardcoded constant in `SettingsViewModel` to a single source of truth in `gradle/libs.versions.toml`. A Gradle task in `shared/build.gradle.kts` will generate an `AppConfig.kt` object containing the version, which the shared module code reads at runtime. The Android app's `versionName` and `versionCode` will also read from `libs.versions.toml`.

## Decisions Made

### 1. Version generation approach: Gradle-generated Kotlin file

- **Decision**: Use a custom Gradle task to generate `AppConfig.kt` into the build directory, registered as a commonMain source set.
- **Rationale**: The `androidMultiplatformLibrary` plugin does not support `buildConfigField`. BuildKonfig is a third-party plugin that adds unnecessary complexity for a single constant. A simple Gradle task that writes a Kotlin file is zero-dependency, easy to understand, and works for both Android and iOS targets.
- **Alternatives considered**:
  - **BuildKonfig plugin** — adds a third-party dependency for a trivial use case.
  - **Compose Resources strings.xml** — Gradle cannot inject values into Compose Resources at build time without custom tooling. Overcomplicating.
  - **expect/actual with platform-specific BuildConfig** — Android has BuildConfig but iOS does not. Would require an asymmetric solution.

### 2. Version format: Semantic versioning with separate versionCode

- **Decision**: Use `app-versionName = "1.0.0"` (semver) and `app-versionCode = "1"` in `libs.versions.toml`. The generated `AppConfig.kt` only includes `VERSION_NAME` since `versionCode` is Android-specific.
- **Rationale**: Semver is industry standard. The version code is only needed for Google Play and has no meaning in the UI. Keeping them separate in the TOML allows independent bumping.

### 3. Generated file location

- **Decision**: Generate into `${buildDir}/generated/source/appconfig/commonMain/kotlin/com/simplevideo/whiteiptv/AppConfig.kt` and register the directory as a commonMain source directory.
- **Rationale**: Generated files belong in the build directory, not in the source tree. This avoids polluting version control and follows Gradle conventions.

## Changes Required

### Modified Files

#### 1. `gradle/libs.versions.toml`
- **What changes**: Add two new entries at the top of `[versions]`:
  ```toml
  app-versionName = "1.0.0"
  app-versionCode = "1"
  ```
- **Why**: Single source of truth for app version.

#### 2. `shared/build.gradle.kts`
- **What changes**: Add a Gradle task that generates `AppConfig.kt` and register the output directory as a commonMain source set. The task reads the version from `libs.versions.app.versionName.get()`.
  ```kotlin
  val generateAppConfig by tasks.registering {
      val versionName = libs.versions.app.versionName.get()
      val outputDir = layout.buildDirectory.dir("generated/source/appconfig/commonMain/kotlin")

      outputs.dir(outputDir)

      doLast {
          val packageDir = outputDir.get().asFile.resolve("com/simplevideo/whiteiptv")
          packageDir.mkdirs()
          packageDir.resolve("AppConfig.kt").writeText(
              """
              |package com.simplevideo.whiteiptv
              |
              |/** Auto-generated by Gradle. Do not edit manually. */
              |object AppConfig {
              |    const val VERSION_NAME: String = "$versionName"
              |}
              |""".trimMargin()
          )
      }
  }
  ```
  Then register the source directory:
  ```kotlin
  kotlin {
      sourceSets {
          commonMain {
              kotlin.srcDir(generateAppConfig.map {
                  layout.buildDirectory.dir("generated/source/appconfig/commonMain/kotlin").get()
              })
          }
      }
  }
  ```
- **Why**: Generates a Kotlin constant from the Gradle version catalog at build time.

#### 3. `androidApp/build.gradle.kts`
- **What changes**: Replace hardcoded `versionCode` and `versionName`:
  ```kotlin
  // Before:
  versionCode = 1
  versionName = "1.0"

  // After:
  versionCode = libs.versions.app.versionCode.get().toInt()
  versionName = libs.versions.app.versionName.get()
  ```
- **Why**: Android app version now reads from the same source of truth.

#### 4. `shared/src/commonMain/kotlin/com/simplevideo/whiteiptv/feature/settings/SettingsViewModel.kt`
- **What changes**:
  - Remove `private const val APP_VERSION = "1.0"` from the companion object (line 126).
  - Replace both usages of `APP_VERSION` with `AppConfig.VERSION_NAME`:
    - Line 33: `appVersion = AppConfig.VERSION_NAME,`
    - Line 103: `appVersion = AppConfig.VERSION_NAME,`
  - Add import: `import com.simplevideo.whiteiptv.AppConfig`
  - If the companion object becomes empty (only `SUPPORT_EMAIL` and `PRIVACY_POLICY_URL` remain), keep it with those two constants.
- **Why**: Version now comes from the generated config instead of a hardcoded string.

#### 5. `shared/src/commonTest/kotlin/com/simplevideo/whiteiptv/feature/settings/SettingsViewModelTest.kt`
- **What changes**:
  - Line 127: Change `assertEquals("1.0", state.appVersion)` to `assertEquals(AppConfig.VERSION_NAME, state.appVersion)`.
  - Add import: `import com.simplevideo.whiteiptv.AppConfig`
- **Why**: Test should not hardcode the version string; it should verify the ViewModel uses the generated value.

The only "new file" is `AppConfig.kt`, which is generated into the build directory by Gradle. It will not exist in version control.

### No Database Changes

Not applicable.

## Implementation Order

1. **Add version entries to `gradle/libs.versions.toml`** — add `app-versionName` and `app-versionCode` to `[versions]`.
2. **Add code generation task to `shared/build.gradle.kts`** — create the `generateAppConfig` task and register the output as a source directory.
3. **Update `androidApp/build.gradle.kts`** — replace hardcoded `versionCode` and `versionName` with catalog reads.
4. **Update `SettingsViewModel.kt`** — replace `APP_VERSION` with `AppConfig.VERSION_NAME`.
5. **Update `SettingsViewModelTest.kt`** — use `AppConfig.VERSION_NAME` instead of hardcoded `"1.0"`.
6. **Build and test** — verify the generated file works and tests pass.

## Testing Strategy

- **Unit test**: The existing `SettingsViewModelTest.init loads default state when no preferences stored` test validates that `state.appVersion` equals the expected version. After the change, it will compare against `AppConfig.VERSION_NAME`, which verifies the ViewModel reads from the generated config.
- **Build verification**: Run `./gradlew :shared:testAndroidHostTest` to confirm the generated file is available during compilation and tests.
- **Android build**: Run `./gradlew :androidApp:assembleDebug` to verify `versionName` / `versionCode` are read correctly.
- **Edge case**: If the Gradle task doesn't run before compilation, the build will fail with a missing class error. The `srcDir(generateAppConfig.map { ... })` dependency ensures Gradle wires the task automatically.

### Coroutine test patterns

No changes needed — the existing test coroutine setup is unaffected by this change.
